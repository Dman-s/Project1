# TT100K 中文标签设计

## 目标

交通标志识别结果不再把 `p10`、`p23`、`pl5` 等 TT100K 内部类别代码直接展示给用户，而是展示对应的中文交通标志含义。模型类别编号和原始代码保持不变。

## 范围

- 覆盖项目统一使用的 45 个 TT100K common 类别。
- 单图、批量和任务历史接口采用同一套中文映射。
- 结果卡片和文本摘要继续优先使用 API 的 `display_name`。
- GTSRB 分类结果沿用现有 43 类中文映射。
- 本次不改变模型权重、检测框坐标、阈值或类别编号。
- 本次不修改标注图片内的英文/代码框文字；用户截图中的结果列表改为中文。

## 数据契约

TT100K 检测结果保持以下字段语义：

- `type`：原始类别代码，例如 `p10`。
- `class_name`：原始类别代码，例如 `p10`。
- `class_name_cn`：真实中文含义，例如 `禁止机动车通行`。
- `display_name`：面向用户的名称，优先等于 `class_name_cn`。
- `class_id`：项目 canonical common-45 类别编号，不变。

这样既不破坏模型、数据库和既有调用方，也能让前端直接展示可读含义。

## 后端设计

在 `tt100k_labels.py` 中维护与 `TT100K_COMMON_45_CLASSES` 等长、同序的中文标签元组，并提供按类别代码查询中文名的函数。映射以 TT100K 自带的 `marks/pad-all` 标志模板、原始标注样本及中国道路交通标志含义为依据，不采用仅凭类别缩写猜测的第三方映射。

`DetectionTaskService._serialize_signs` 根据识别模式选择中文标签：GTSRB 分类使用现有 `GTSRB_LABELS_ZH`，TT100K 检测使用新增映射。数据库继续保存原始 `class_name`，同时保存 `class_name_cn`。

任务历史序列化在旧记录的 `class_name_cn` 为空时，根据 `class_name` 动态补充 TT100K 中文名，使已经存在的检测任务也能返回可读标签，无需数据库迁移。

检测场景注册信息同步保存 TT100K 的代码到中文名映射，保证场景元数据与检测结果一致。

## 前端设计

现有结果卡片和文本摘要已经按照 `display_name -> class_name -> type` 的顺序选择标签。后端补齐 `display_name` 后，截图中的 `p10 / p23 / pl5` 会自动变为中文，无需复制一份前端映射表。

不在前端维护类别翻译，避免单图、批量、任务历史和其他消费者之间出现不同版本。

## 错误处理

如果遇到映射表之外的类别代码，API 回退到原始代码，不中断识别任务。测试会强制 common-45 代码与中文标签数量、顺序完整对应，防止新增类别时静默遗漏。

## 测试与验收

- 标签单元测试验证 45 类映射完整且 `p10`、`p23`、`pl5` 等代表类别含义正确。
- 检测任务测试验证新识别结果同时保留原始代码并返回中文 `display_name`。
- 任务历史 API 测试验证旧记录即使没有 `class_name_cn` 也能动态返回中文名。
- GTSRB 测试保持通过，确认分类标签没有回退。
- 后端全套测试、前端测试与生产构建通过。
- 用截图对应的 `train/1395.jpg` 重新走真实 API，确认结果卡片数据返回中文标签。
